  <action>
    This task requires actions in two external dashboards that Claude cannot automate. Complete both steps before resuming.

    **Step 1 — Create Upstash Redis database:**
    1. Visit https://console.upstash.com and log in (or create a free account)
    2. Click "New Database"
    3. Name: `nightlight-tales` (or any name you prefer)
    4. Type: Regional
    5. Region: Choose the region closest to your Vercel deployment (us-east-1 is a safe default)
    6. Plan: Free tier is sufficient for this use case
    7. Click "Create"
    8. Once created, open the database and click the "REST API" tab
    9. Copy the **REST URL** (format: `https://XXXX.upstash.io`) — this is `UPSTASH_REDIS_REST_URL`
    10. Copy the **REST Token** (long base64 string) — this is `UPSTASH_REDIS_REST_TOKEN`

    **Step 2 — Add credentials to Vercel:**
    1. Visit https://vercel.com and open your project (nightlight-tales / bed-time-nu)
    2. Go to Settings → Environment Variables
    3. Add first variable:
       - Key: `UPSTASH_REDIS_REST_URL`
       - Value: (paste the REST URL from Step 1)
       - Environment: Production (check only Production — local dev uses the bypass)
    4. Add second variable:
       - Key: `UPSTASH_REDIS_REST_TOKEN`
       - Value: (paste the REST Token from Step 1)
       - Environment: Production (check only Production)
    5. Click Save for each variable

    **Step 3 — Trigger a redeployment:**
    Vercel environment variable changes only take effect on new deployments. Either:
    - Push a trivial commit (e.g. add a comment) to trigger auto-deploy, OR
    - Go to Vercel Dashboard → Deployments → click the latest deployment → "Redeploy"

    **Known behavior (Upstash free tier):** Free databases pause after inactivity. The first request after a pause may be slightly slower (Redis resume latency). The `@upstash/ratelimit` library allows requests through on timeout by default — this means the first request after a cold resume is not incorrectly rate-limited.

    **Automated preparation — Claude generates the curl loop command:**
    ```bash
    # Send 11 POST requests to production /api/generate from the same IP.
    # Requests 1-10 should return 200 (or 400 if Claude API fails — that's fine).
    # Request 11 should return 429.

    for i in $(seq 1 11); do
      echo "Request $i:"
      curl -s -o /dev/null -w "%{http_code}" \
        -X POST https://bed-time-nu.vercel.app/api/generate \
        -H "Content-Type: application/json" \
        -d '{"name":"Test","age":5,"theme":"Animals","duration":3}'
      echo ""
      sleep 0.5
    done
    ```

    Run this command in your terminal. Expected output:
    ```
    Request 1: 200
    Request 2: 200
    ...
    Request 10: 200
    Request 11: 429
    ```

    Note: Requests may return 200 (story generated), 400 (validation error — unlikely with valid payload), or 500 (Claude API error). Any non-429 for requests 1-10 is acceptable. Request 11 MUST be 429.

    **Alternative quick check (if you've recently made requests):**
    ```bash
    curl -s -X POST https://bed-time-nu.vercel.app/api/generate \
      -H "Content-Type: application/json" \
      -d '{"name":"Test","age":5,"theme":"Animals","duration":3}' | head -c 200
    ```
    If you get `{"error":"You've created a few stories recently. Try again in a bit."}` with status 429, the rate limit is enforcing.
